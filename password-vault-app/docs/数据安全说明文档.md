# 密拾 APP 数据安全说明文档

本文面向用户与审核人员，说明密拾如何保护账号、密码、网址与截图数据：**加密算法、密钥派生、存储位置、备份、卸载与隔离策略**。

技术栈：Capacitor WebView（`www/index.html` + `www/js/mishi-ext.js`）+ Android 原生层（截图与悬浮窗）。

---

## 1. 安全设计原则

1. **本地优先**：无云端账号、无自动同步、Manifest 无 `INTERNET`。  
2. **解锁前不可读**：vault  ciphertext 保存在应用私有存储；须正确 PIN 才能解密。  
3. **分层存储**：Web 加密库 + 原生 pending 队列/截图文件 + SharedPreferences 非敏感配置。  
4. **备份可选且可加密**：推荐 `.mishi` 格式；用户自行保管备份密码。

---

## 2. 主库加密：AES-256-GCM

### 2.1 算法与实现

- **算法**：AES-GCM，256 位密钥（Web Crypto `crypto.subtle`）。  
- **IV**：每次加密随机 12 字节，与密文一并存储。  
- **明文内容**：JSON 序列化的 `entries` 数组（账号、密码、网址、备注、截图 base64 等）。

相关逻辑见 `www/index.html` 中 `encrypt()` / `decrypt()` / `saveVault()` / `loadVault()`。

### 2.2 密钥派生：PBKDF2

- **用户 PIN**（4–6 位数字）**不**以明文写入存储。  
- **盐值**：16 字节随机 `salt`，与 vault 元数据同存。  
- **KDF**：PBKDF2-HMAC-SHA256，**100 000** 次迭代，派生 AES-256-GCM 密钥。  
- **PIN 校验**：单独保存 `pinHash`（PIN + 固定后缀经 SHA-256），用于解锁比对，**不能**反推 PIN。

忘记 PIN：**无法恢复** vault（符合任务书风险提示）。

### 2.3 存储位置

- 密文与元数据：Capacitor WebView **`localStorage`** 键 `vault_v1`（位于应用私有目录，其他 App 无法访问）。  
- **未**使用明文 SQLite 存密码字段。

## 3. 备份加密（.mishi）

设置 →「密拾 .mishi 备份」（`mishi-ext.js`）：

| 项目 | 说明 |
|------|------|
| 格式标识 | `mishi-backup-v1` |
| 备份密码 | 与 PIN **独立**；用户导出/导入时输入 |
| KDF | PBKDF2-HMAC-SHA256，**120 000** 次迭代 + 随机 16 字节 salt |
| 内容加密 | AES-256-GCM 包裹 `{ meta, entries }` |
| 文件扩展名 | `.mishi`（Base64 包裹的 JSON 描述符） |

导入支持**合并**或**替换**现有 entries。

> ⚠️ 设置中仍存在**明文 JSON** 导入/导出（历史功能）。明文备份含全部密码与截图数据，**强烈建议仅使用 .mishi**；后续版本应移除或强制二次确认。

## 4. 截图与隔离

### 4.1 流水线

1. **监听**（可选）：`OverlayService` 检测 MediaStore 新截图。  
2. **复制**：`ScreenshotStore.copyToVault` → `files/vault_shots/<uuid>.jpg`（应用私有）。  
3. **待处理队列**：SharedPreferences `pending_screenshots` 记录路径。  
4. **入库**：用户解锁 APP 后 `consumeScreenshot` → base64 写入加密 `entries` → 删除 pending 文件。  
5. **系统相册**：收纳成功后尝试 `MediaDeleteHelper` 删除原截图（取决于系统/厂商权限）。

### 4.2 隔离性

- 截图在入库前仅存在于**应用 sandbox**；不写入公开 DCIM 目录（除非用户主动「导出到相册」）。  
- 入库后截图与其它条目一样处于 **AES-GCM vault** 内。  
- ⚠️ **pending 阶段** JPEG 为文件系统明文，保护依赖「未解锁时其它 App 仍无法跨 sandbox 读取」；设备 root 或备份整个应用数据时风险升高。

### 4.3 悬浮窗搜索索引

- 解锁后同步到原生的 `vault_search_index` **仅含**名称/账号/网址等检索字段（`buildSearchIndex` **不含** password）。  
- 设计意图：避免在 SharedPreferences 长期存放明文密码。  
- 副作用：悬浮窗「快速搜密码」当前与索引不一致，需产品/开发后续修复（见功能核对清单）。

---

## 5. 悬浮窗与 pending 录入

- 悬浮窗新增的账号/书签写入 `pending_vault_entries`（JSON，**含明文密码直至合并**）。  
- 用户打开并解锁 APP 后 `consumePendingEntries` 合并进加密 vault 并清空队列。  
- **建议**：合并后依赖 vault 加密；缩短 pending 停留时间，避免长期未打开 APP。

临时剪贴板：`VaultCache` 存文本 + 过期时间（默认 5 分钟），到期清空。

## 6. 隐私防护功能

| 功能 | 机制 |
|------|------|
| 自动锁定 | 无操作超时调用 `lockApp()`，清除内存中 `cryptoKey` |
| 切后台模糊 | `#privacy-blur` 覆盖界面，降低偷窥风险 |
| 活动日志 | 仅记录操作类型与时间片段，不记录完整密码 |

生物识别解锁：**当前未实现**。

---

## 7. 卸载与数据销毁

- Android 卸载应用会删除其私有目录（含 WebView storage、`vault_shots`、SharedPreferences）。  
- **无**远程「注销账号」；数据是否曾导出为 `.mishi` 由用户自行保管。  
- Manifest 中 `android:allowBackup="true"` 可能允许系统级全量备份到 Google 账号（视设备设置而定）。若需与「卸载即不可恢复」营销口径严格一致，应评估改为 `false` 并文档说明。

应用内「清空数据」会 wipe 本地 vault（需 PIN/确认，见设置页）。

---

## 8. 威胁模型（简要）

| 威胁 | 缓解 |
|------|------|
| 其它 App 读取文件 | Android sandbox |
| 明文抓包上传 | 无 INTERNET |
| 丢失手机 | PIN + 全量加密；防暴力依赖设备锁屏 |
| 忘记 PIN | 无法解密（by design） |
| 恶意 root / 物理取证 | 无法完全保证；敏感 pending 文件与内存密钥仍可能被提取 |
| 明文 JSON 备份泄露 | 用户行为风险；应使用 .mishi |

---

## 9. 合规与审核要点

- 数据处理**本地化**；权限用途与《权限说明文档.md》一致。  
- 向用户明确：**忘记 PIN / 备份密码不可找回**。  
- 截图功能说明：读取媒体库仅为识别截图，非全库上传。
